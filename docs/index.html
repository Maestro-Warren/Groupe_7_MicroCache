MicroCache

Cache haute performance pour billetterie concert - démonstration d'un système de cache distribué validant le gain de performance dans un scénario de charge élevée.

Voir l'équipe Détails du projet

Vue d'ensemble

MicroCache est une démonstration complète d'un système de cache distribué conçu pour améliorer les performances dans des scénarios de charge élevée, comme la vente de tickets de concert.

Serveur MicroCache

Cache clé/valeur en mémoire TCP thread-safe écrit en Rust avec Tokio pour des performances optimales.

API Backend

API REST Node.js avec Express implémentant le pattern cache-aside pour la billetterie concert.

Dashboard Temps Réel

Interface web avec visualisation en temps réel des métriques via WebSocket et Chart.js.

Plus de détails sur le projet

MicroCache combine une architecture Rust/Node.js avec un dashboard en temps réel pour illustrer les bénéfices concrets du cache distribué. Le système couvre le cycle complet : requête API, consultation du cache, fallback sur la base de données, et rebond vers le dashboard.

Architecture

Architecture modulaire et scalable du système MicroCache

graph TD A[Dashboard
Port 3000] --> B[API Backend
Node.js Express] B --> C[SQLite DB] B --> D[MicroCache TCP
Port 6379] D --> E[Rust Tokio Server] E --> F[Store Thread-safe] E --> G[Reaper Background GC] E --> H[Persistence Snapshots]

Backend Architecture

  • Cache-Aside Pattern: L'API vérifie d'abord le cache avant la base de données
  • WebSocket Metrics: Diffusion temps réel des métriques toutes les 500ms
  • SQLite Database: Stockage persistant des données de concert
  • Metrics Collector: Collecte des statistiques avec fenêtres glissantes

Rust MicroCache

  • Tokio Async: Serveur TCP asynchrone haute performance
  • Thread-Safe Store: Structure de données concurrente avec TTL
  • Background Reaper: Nettoyage automatique des entrées expirées
  • Binary Persistence: Snapshots sur disque pour la durabilité

Installation & Configuration

Guide complet pour démarrer avec MicroCache

Prérequis

  • Rust 1.70+ (avec Cargo)
  • Node.js 18+ (avec npm)
  • SQLite (inclus avec Node.js experimental)

Installation

1. Cloner le projet

git clone https://github.com/Maestro-Warren/Groupe_7_MicroCache

2. Installer les dépendances API

cd microcache-demo/api-server && npm install

3. Compiler MicroCache

cd Groupe_7_MicroCache && cargo build --release

Démarrage Rapide

1

Lancer MicroCache

cargo run
2

Lancer l'API

npm start
3

Ouvrir le Dashboard

http://localhost:3000

Utilisation

Comment utiliser MicroCache dans votre application

Protocole MicroCache

Protocole textuel ligne-par-ligne (terminateur: \n)

PING → PONG
SET key value → OK
SET key value EX 30 → OK
GET key → value ou NIL
EXISTS key → 1 ou 0
DEL key → OK

Client TCP

Connexion au serveur MicroCache sur le port 6379

// Exemple de connexion en Node.js
const net = require('net');

const client = net.createConnection({ port: 6379 }, () => {
  client.write('SET concert:1:tickets 150\\n');
  client.write('GET concert:1:tickets\\n');
});

client.on('data', (data) => {
  console.log(data.toString());
});

API REST

Endpoints de l'API de billetterie concert

GET

/api/concerts

Récupère la liste de tous les concerts

Réponse: Liste des concerts avec ID, nom, places disponibles
GET

/api/tickets?concertId={id}

Récupère les places disponibles pour un concert (avec cache)

Réponse: Nombre de places disponibles (cache-aside pattern)
GET

/api/tickets/nocache?concertId={id}

Récupère les places disponibles sans cache (comparaison)

Réponse: Nombre de places disponibles (directement depuis DB)
GET

/api/stats

Récupère les métriques de performance

Réponse: Requêtes totales, hit rate, latence moyenne, speedup factor

Script de Test de Charge

Outil intégré pour benchmarker les performances

# Défaut : 100 req/route, 10 concurrentes
node src/loadtest.js

# Personnalisé
node src/loadtest.js --requests 500 --concurrency 50 --concertId 4 --duration 30

Options du Test

  • --requests: Nombre total de requêtes (défaut: 100)
  • --concurrency: Nombre de requêtes simultanées (défaut: 10)
  • --concertId: ID du concert à tester (défaut: 1)
  • --duration: Durée du test en secondes (défaut: 10)
