Vue d'ensemble du projet MicroCache

Une présentation complète du projet MicroCache : objectifs, architecture globale, composants et gains attendus dans un contexte de billetterie à forte charge.

Objectif principal

MicroCache vise à démontrer l’impact d’un cache distribué dans une application de billetterie concert. Le but est de réduire les accès directs à la base de données, d’accélérer les réponses et de maintenir une haute disponibilité en charge.

  • Cache en mémoire : stockage clé/valeur TCP avec support TTL.
  • API billetterie : route d’accès aux places et comparaison cache/non-cache.
  • Dashboard temps réel : métriques, hit rate et latence en direct.
  • Tests de charge : script de benchmark pour mesurer les gains.

Pourquoi un cache distribué ?

Dans les systèmes modernes, les accès à la base de données représentent souvent le goulot d'étranglement. Un cache distribué permet :

  • de diminuer fortement la latence de réponse,
  • d’augmenter le throughput,
  • de décharger la base de données sous pic de trafic,
  • de maintenir une expérience utilisateur fluide.

Composants clés

Serveur Rust

Module principal qui gère les commandes TCP, le store concurrent et le nettoyage TTL.

API Node.js

Interface entre le client HTTP et MicroCache, avec cache-aside et métriques temps réel.

Dashboard

Visualisation en direct des performances, latence, hit rate et comparaison cache/non-cache.
